# topix1000 Export Disclosure Provider

## Purpose

`Topix1000ExportProvider` (`src/jp_stock_analysis/providers/topix1000_export.py`)
reads deterministic EDINET disclosure exports generated by
**topix1000_disclosure_platform** and maps them into this engine's existing
`DisclosureDocument` flow.

Division of responsibilities:

- **topix1000_disclosure_platform** is the ingestion platform: it calls the
  EDINET API, stores raw archives and parsed XBRL facts in PostgreSQL, and
  exports deterministic JSON files.
- **jp-stock-analysis-engine** is the analysis engine: it reads those export
  files as an optional, file-based disclosure provider. It has **no PostgreSQL
  dependency**, **no direct database access**, and **never calls EDINET
  live**.

The default disclosure source remains local `<ticker>.txt` files; this
provider is explicit opt-in via `--disclosure-provider topix1000-export`.

## Expected export directory layout

```
export_dir/
├── index.json
└── disclosures/
    ├── S100XXXX.json
    └── S100YYYY.json
```

`index.json` is either an object with a `documents` list or a bare list.
Each entry provides a `path` (relative to the export dir) and/or a `doc_id`
(which defaults the path to `disclosures/<doc_id>.json`):

```json
{
  "documents": [
    {"doc_id": "S100XXXX", "path": "disclosures/S100XXXX.json"}
  ]
}
```

## Per-document schema example

```json
{
  "doc_id": "S100XXXX",
  "target_date": "2026-06-04",
  "edinet_code": "E00001",
  "sec_code": "72030",
  "ticker": null,
  "company_name": "サンプル自動車株式会社",
  "filing_type_key": "annual_securities_report",
  "document_type": "annual_securities_report",
  "form_code": "030000",
  "doc_type_code": "120",
  "fiscal_year": 2025,
  "source_paths": {
    "original_zip": "raw/S100XXXX/original.zip",
    "document_pdf": "raw/S100XXXX/document.pdf",
    "csv_zip": "raw/S100XXXX/csv.zip",
    "list_response_json": "raw/2026-06-04/list_response.json"
  },
  "xbrl_facts_summary": {"facts_count": 1234, "contexts_count": 56, "units_count": 7},
  "text": "当連結会計年度は…（extracted narrative text, or null）"
}
```

Unknown fields are tolerated and ignored.

## Mapping into `DisclosureDocument`

- ticker key: `ticker` if present, else `sec_code` (a 5-digit EDINET
  `secCode` such as `72030` is normalized to the 4-digit listing code
  `7203`), else `edinet_code` as a last-resort key with a warning.
- `doc_id`, `edinet_code`, `company_name`, `document_type` (falls back to
  `filing_type_key`), and `fiscal_year` map to the same-named
  `DisclosureDocument` fields.
- `target_date`, `filing_type_key`, `form_code`, `doc_type_code`, every
  `source_paths.*` entry (as `source_path.<name>`), and every
  `xbrl_facts_summary.*` entry (as `xbrl.<name>`) are preserved in
  `source_metadata` and propagate into the disclosure analysis result and
  JSON report.
- If `text` is a non-empty string it flows into the rule-based disclosure
  NLP exactly like a local TXT disclosure.
- If `text` is `null`/missing, the provider builds a **metadata-only**
  document with an explicit warning; the analyzer then reports zero findings
  and `confidence_score 0.0`. **Text is never fabricated.**
- If one ticker has several documents, the latest (`target_date`, then
  `doc_id`) is analyzed and a warning lists the doc ids set aside.

## CLI usage

```bash
python -m jp_stock_analysis.cli analyze \
    --prices tests/fixtures/prices_sample.csv \
    --fundamentals tests/fixtures/fundamentals_sample.csv \
    --metadata tests/fixtures/company_metadata_sample.csv \
    --disclosure-provider topix1000-export \
    --topix1000-export-dir /path/to/export_dir \
    --output-dir /tmp/out
```

- `--disclosure-provider local|topix1000-export` (default `local`; the
  existing `--disclosures` TXT-directory behavior is unchanged).
- `--topix1000-export-dir PATH` is required with `topix1000-export` and
  cannot be combined with `--disclosures`.
- Works with any price provider (`--provider local|jquants-cache|jquants-live`);
  disclosure source and price source are orthogonal.
- `--signal-mode` defaults to `analysis_only` as always; `trade_signal`
  remains explicit opt-in and the report disclaimer is unchanged.

## Error behavior

| Condition | Behavior |
|---|---|
| Export dir or `index.json` missing | `ProviderError` with the offending path |
| Invalid JSON (index or document) | `ProviderError` naming the file |
| Index entry without `path`/`doc_id` | `ProviderError` with the entry position |
| Per-document file missing | `ProviderError` with the path |
| Document with no `ticker`/`sec_code`/`edinet_code` | `ProviderError` |
| Empty `documents` list | warning on stderr, empty result (no crash) |
| `text` null/missing | metadata-only document, warning, confidence 0 |

## Limitations

- The provider does **not** extract text from PDF/XBRL archives; it only
  consumes text already present in the export. Text extraction belongs to
  the ingestion platform.
- `source_paths` are recorded as provenance strings only; the engine never
  opens the referenced archives.
- `xbrl_facts_summary` and `key_facts_sample` are not used as fundamentals;
  financial statements still come from the existing CSV/J-Quants providers.
- One document per ticker is analyzed per run (the engine's disclosure flow
  is single-document); use `get_disclosures(ticker)` for the full list.
- No RAG, no broker logic, no auto-trading. Valuation alone never produces a
  buy signal; all existing safety behavior is unchanged.
